# AGENTS.md — Code Generation Agent for **Dict-ai-te (Dioxus)**

## Agent Objective

Create a **fourth** version of the Dict-ai-te application implemented in **Rust + Dioxus** that preserves **all** functional behavior of the current Python (desktop and web) and Rust (egui) versions: audio recording and file import, transcription, translation, text-to-speech playback, and any auxiliary features defined in the existing documentation and source code. The **UI must match the Python web interface exactly** in layout, interaction flow, and behavior.

## Non-Negotiable Requirements

1. **UI parity is critical.** The Dioxus UI must replicate the Python **web** interface pixel-for-pixel where feasible and interaction-for-interaction otherwise. Do not “improve” or “modernize” the UX unless parity requires it.
2. **Feature parity:**

   * Audio capture and upload
   * Transcription
   * Translation
   * Audio playback (TTS or wave playback as implemented today)
   * Any project-specific settings, languages, hotkeys, batching, export, or logging present in existing apps
3. **Source tree reorganization:**

   * Move the current Rust egui app to **`src/dict-ai-te-egui/`** as its **own crate**.
   * Create a new Dioxus app at **`src/dict-ai-te-dioxus/`** as its **own crate**.
   * Convert the repository into a **Cargo workspace** so each Rust app has an independent `Cargo.toml` and can compile separately.
   * The existing single `Cargo.toml` must be refactored accordingly.
4. **Do not change Python sources** except where strictly necessary to expose stable APIs the Dioxus app already relies on.
5. **Keep naming and paths exactly as specified**: `src/dict-ai-te-egui` and `src/dict-ai-te-dioxus`.

## Repository Assumptions the Agent Must Discover

* The Python implementations live under a `Dict-ai-te/` directory or module. There are two versions: **desktop** and **web**.
* The Rust egui version lives in the current Rust source directory.
* There may already be reusable Rust library code for audio IO, HTTP client calls, and model bindings.
* The Python web app exposes endpoints or has a server layer for transcription, translation, and TTS. If not, the Rust versions have equivalent local functionality. The agent must discover the actual integration path and mirror it.

## Deliverables

1. **New crate** at `src/dict-ai-te-dioxus/` with a working Dioxus app that achieves full feature parity and UI parity with the Python web version.
2. **Relocated egui crate** at `src/dict-ai-te-egui/`.
3. **Top-level Cargo workspace** with `Cargo.toml` at repo root and per-crate `Cargo.toml` files.
4. **Build and run docs** added or updated in `README.md` with exact commands for both Rust apps.
5. **Parity checklist** documented in `docs/dioxus-parity-checklist.md` demonstrating each feature and UI element matches the Python web interface.

## Implementation Instructions

### 1) Discover and lock the canonical UX

* Inspect the Python **web** app templates, CSS, and JS.
* Extract layout, component hierarchy, control labels, validation, keyboard shortcuts, progress indicators, error states, spinners, and notifications.
* Capture exact spacing, typography, and color references used by the web app.
* Use a style adapter layer if needed to apply the same CSS within Dioxus or re-implement in Rust with equivalent styles. The result must look and behave the same.

### 2) Dioxus UI structure

* Build components to mirror the Python web pages:

  * **Top bar** with recording controls and status
  * **Language selectors** for source and target
  * **Mode toggles** for transcription vs translation
  * **File-drop zone** with progress
  * **Transcript panel** with timestamps if present
  * **Playback controls** for TTS or audio output
  * **Settings drawer** if the web app has one
* Preserve route structure and navigation if the web app is multi-page. Otherwise build a single-page layout with equivalent sections.
* Ensure the **Start Recording / Stop Recording** behavior matches exactly. If the Python web version auto-submits after stop, the Dioxus app must do the same.

### 3) Audio, transcription, translation, playback

* Prefer reusing the **same integration surface** as the current Rust or Python apps. Two valid strategies:

  1. **HTTP API reuse** — If the Python web server exposes `/transcribe`, `/translate`, `/tts` endpoints, call them from Dioxus using the same request and response schema.
  2. **Local Rust bindings** — If the Rust egui app uses local bindings to engines (e.g., Whisper, Vosk, Coqui, or vendor SDKs), extract common logic into a shared Rust library crate inside the workspace and link it from both Rust apps.
* The agent must detect which integration exists and implement accordingly. Do not invent new protocols if stable ones exist.
* Audio recording on desktop targets requires appropriate renderer selection (e.g., `dioxus-desktop` with audio capture or a small native helper). For web targets, use Web APIs via Dioxus web renderer. Provide a thin abstraction so both targets share the same UI and logic.

### 4) Build targets

* Implement **desktop** and **web** targets for the Dioxus app if feasible within the project constraints. Otherwise implement the target that guarantees parity with the Python **web** UI first, then add desktop.
* Provide `Makefile` or Cargo aliases for:

  * `cargo run -p dict-ai-te-dioxus` (desktop)
  * `dx serve` or `trunk serve` for web if using Dioxus web toolchain
  * `cargo run -p dict-ai-te-egui` for the egui app

### 5) Error handling and logging

* Match the Python web app’s error flows and messages. Show the same user-visible strings where applicable.
* Log equivalent events at similar verbosity in Rust.

### 6) Tests and verification

* Add minimal UI smoke tests if applicable.
* Add API contract tests to verify request and response parity with the Python server if using HTTP.
* Create `docs/dioxus-parity-checklist.md` listing each feature and UI element with a pass/fail box. Fill it out as you implement.
