System overview

dict-ai-te is a live dictation and live translation application built around OpenAI Realtime.

The user presses a button, speaks into the microphone, and sees transcription or translation accumulate in real time. Once the session ends, the transcript can be saved, copied, or played back through TTS.

All three implementations connect to the same external service: the OpenAI Realtime API. There is no local speech recognition model; the STT and translation pipeline runs on OpenAI's infrastructure.

The application runs in two session modes selected before recording starts: transcription connects to the transcription intent endpoint with gpt-4o-transcribe, while translation uses the realtime endpoint with gpt-realtime.

Repository layout

The repository is split into a Rust application, a shared Python core, and UI packages.

The Rust implementation lives under src/, the Python shared logic lives in dictaite_core/, and the Python UI layers live in dictaite/. The shared settings file keeps the implementations aligned across frontends.

Key alignment points

  • Realtime transport, event parsing, transcript assembly, and audio conversion have matching Rust and Python implementations.
  • The settings format is shared and migrated from legacy TOML to JSON.
  • The GTK and web frontends are thin shells around the same shared runtime concepts.

Audio pipeline

Every frontend converges on the same chunk format before the socket send.

The Rust implementation is the most explicit about threading and conversion, while the Python GTK and web variants mirror the same logical flow with different APIs. The relevant invariant is stable: mono, 24 kHz, PCM16, base64-encoded chunks.

Rust

Threaded capture and conversion

cpal callback
  └─ samples (f32)
      └─ bounded channel
          └─ audio worker thread
              ├─ downmix to mono
              ├─ resample to 24 kHz
              ├─ buffer 40 ms chunks
              ├─ convert to PCM16 LE
              └─ base64 encode + send
Python GTK

Direct 24 kHz mono capture

sounddevice.InputStream(24_000, channels=1)
  └─ frames already at target rate
      └─ normalize_audio
      └─ float32 → PCM16
      └─ base64 encode
      └─ queue.put_nowait(chunk)
      └─ asyncio worker reads queue
Web

Browser capture and server relay

Browser audio graph
  └─ downsample to 24 kHz
  └─ float → i16
  └─ base64 encode
  └─ WebSocket { type: "audio" }
Server
  └─ forwards chunks to OpenAI Realtime
OpenAI Realtime sessions

Transcription and translation share the same lifecycle, then diverge on session intent and output handling.

Transcription session

gpt-4o-transcribe

Connects to the transcription intent endpoint, sends a session update with server VAD and optional source language, and commits the buffer on stop.

wss://api.openai.com/v1/realtime?intent=transcription
Authorization: Bearer <key>

session.update
  type: "transcription"
  audio.input.format.rate = 24000
  transcription.model = "gpt-4o-transcribe"

Translation session

gpt-realtime

Uses the realtime endpoint, instructs the model to return only translated text, and sets output modalities to text. Source transcription and translation can accumulate together.

wss://api.openai.com/v1/realtime?model=gpt-realtime

session.update
  type: "realtime"
  output_modalities: ["text"]
  instructions: "Translate the user's speech into <target>..."

Transcript assembly

Segments preserve order, completion state, and anonymous fragments.

The assembler handles out-of-order deltas and completions while preserving insertion order. Rust keeps a separate order list because its map is keyed; Python uses insertion-ordered structures directly. Both join non-empty segments in order, then append anonymous fragments.

Rust structure
TranscriptAssembler {
  order: Vec<String>
  segments: BTreeMap<String, Segment>
  anonymous: Vec<String>
}

Segment { text, final_text }

The order list keeps the first-seen sequence stable even though the map itself is alphabetically keyed.

Python structure
TranscriptAssembler {
  _segments: OrderedDict[str, _Segment]
  _anonymous: list[str]
}

complete() appends anonymous completions unconditionally

The Python version matches the Rust semantics while relying on insertion ordering in the underlying dictionary.

Text-to-speech and playback

TTS is a separate path, but it reuses the same output text and voice settings.

Rust caches the decoded clip so repeated playback can skip the network call, while the GTK and web paths fetch fresh audio each time. The user-facing outcome remains the same: speak the current transcript aloud in the selected voice.

Rust path

Uses a blocking reqwest client, decodes WAV or MP3 bytes into AudioClip, and keeps a cached clip for repeated playback with the same voice.

Python GTK and web

GTK plays the returned WAV bytes through sounddevice and soundfile. The web frontend asks the browser to play the WAV response with a normal audio element.

Settings and configuration

A single JSON settings file keeps the implementations aligned.

All three apps use ~/.dictaite/settings.json. Legacy TOML settings are migrated transparently on first load, and the config directory can be overridden with DICTAITE_HOME.

Shared settings format

{
  "default_language": null,
  "translate_by_default": false,
  "default_target_language": "en",
  "female_voice": "nova",
  "male_voice": "onyx"
}
Rust

Settings live in a struct with optional language fields and lowercase voice names normalized by fill_defaults.

Python

The dataclass mirrors the Rust shape and serializes with pretty JSON, keeping the frontends interoperable.

Migration

Legacy TOML is parsed and migrated to JSON without deleting the older file.

Override

DICTAITE_HOME redirects the config directory for all implementations.

UI layers

The UI implementations share behavior but differ in presentation model and runtime mechanics.

Rust egui

Immediate-mode application

The entire interface is rebuilt every frame from the DictaiteApp state machine. It includes explicit state for live capture, event channels, transcript text, TTS playback, and the settings modal.

Python GTK 4

Retained-mode window

Gtk.ApplicationWindow keeps a persistent widget tree. Background threads push updates back via GLib.idle_add for audio playback, timers, and live session events.

Python Flask / web

Thin server, browser UI

The server proxies audio to OpenAI Realtime and returns normalized events to the browser. Audio capture and playback are owned by browser APIs.

End-to-end data flow

The transcription and translation paths are the same until the event vocabulary diverges.

Transcription flow

Microphone samples are downmixed, resampled, converted to PCM16, and base64-encoded before reaching the transcription endpoint. The UI updates from SourceDelta and SourceCompleted events.

Translation flow

The translation endpoint emits translated text deltas in addition to source transcript updates. The active transcript becomes the translated transcript while recording is live.

Architectural watchpoint

The user-visible behavior is aligned, but the implementations still differ in capture APIs, scheduling, and playback mechanisms. When editing one frontend, it is worth checking the corresponding shared module before assuming parity already exists.

Operational implication

Because the settings file and event vocabulary are shared, most cross-frontend changes should start in the shared core, then fan out to the UI shells only where they add rendering or input-specific behavior.
